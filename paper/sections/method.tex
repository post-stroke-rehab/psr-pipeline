\section{Method}

\subsection{Overview}

\repo\ implements a modular decoding workflow that starts from raw multichannel sEMG and gesture labels and ends with either five sigmoid-compatible logits or downstream evaluation artifacts. The method is easiest to read as four connected bands: data harmonization and ingestion, signal processing and feature extraction, model fitting, and evaluation or deployment-facing analysis. Figure~\ref{fig:pipeline} summarizes the repository pipeline and the open-extension details incorporated into this updated paper revision.

\subsection{Data harmonization and ingestion}

Before filtering begins, the repository aligns gesture labels with raw PhysioMio recordings, structures subject-level examples, performs deterministic patient-level splitting, and caches processed tensors for reuse. These ingest operations matter because they define the effective task boundary: which arm is selected, how segment labels are mapped into five binary finger targets, and whether train-test leakage is avoided by splitting patients rather than windows. The updated paper figure makes these steps explicit because they are distributed across loaders and processing utilities rather than hidden inside one monolithic script.

\subsection{Signal preprocessing}

The preprocessing module is explicitly parameterized by a frozen configuration object in the repository. Raw sEMG first passes through a fourth-order Butterworth band-pass filter with 20--450 Hz cutoffs and zero-phase forward-backward application. This stage suppresses low-frequency motion artifacts and high-frequency electrical noise while preserving time alignment across channels.

The filtered signal is then denoised with wavelet soft-thresholding. The default configuration uses a Symlet-4 basis, four decomposition levels, universal threshold selection, and a median-based noise estimate. Although the repository does not yet include an ablation isolating this choice, the implementation makes the denoising rule explicit and reproducible.

After denoising, the continuous recording is segmented into 200 ms windows with 50\% overlap. These window settings balance temporal responsiveness with a sufficient sample count for stable summary statistics. The repository disables zero-padding by default, so each window represents observed signal rather than synthetic boundary content.

\subsection{Window-level feature representation}

Each channel-window pair is summarized by twelve handcrafted descriptors spanning time and frequency domains. The time-domain set includes RMS, mean absolute value, integrated EMG, waveform length, variance, zero crossings, slope sign changes, and Willison amplitude. The frequency-domain set includes mean frequency, median frequency, spectral entropy, and total power, with spectral quantities estimated from Welch periodograms~\citep{welch1967fft}. The resulting tensor has shape $(C, W, F)$, where $C$ is the number of channels, $W$ is the number of windows, and $F=12$ is the feature count.

\subsection{Tensor formatting and shared adapter}

The repository standardizes cross-model comparison with one shared adapter. It transforms $(C, W, F)$ tensors into $(N, W, C \times F)$ sequences, where windows act as time steps and all channel features are concatenated into one vector per step. This small interface decision matters because it allows recurrent, convolutional, and graph-based predictors to consume the same derived representation without duplicating preprocessing code. For the CNN branch, the baseline path then permutes the tensor into a channels-first layout, while open PR \#56 makes that step more explicit by flattening the $64 \times 12$ channel-feature grid into 768 one-dimensional input channels for modular student and teacher training~\citep{psrpr56cnn2026}.

\subsection{Predictor families}

The LSTM branch applies two stacked recurrent layers with hidden sizes 128 and 256, followed by a 128-unit fully connected layer and dropout before five output logits. This branch emphasizes temporal accumulation across windows and, in the saved metrics, yields the strongest exact-match accuracy, finger accuracy, and AUROC.

The CNN branch wraps several 1D convolutional student models behind a channels-first permutation layer. In the current training code, the benchmarked CNN consumes the shared window-level sequence after permutation from $(N, W, C \times F)$ to $(N, C \times F, W)$. The repository supports multiple model sizes, but the saved benchmark artifacts are presented as one CNN baseline rather than a full architecture sweep. Open PR \#56 restructures this branch into separate student, teacher, training, distillation, and optimization modules, and open PR \#59 extends it with teacher ensembles plus compact Nano and Micro student variants aimed at edge deployment~\citep{psrpr56cnn2026,psrpr59distill2026}.

The GNN branch interprets each time window as a node in a graph. The implementation supports GCN, GraphSAGE, and GAT operators~\citep{kipf2017gcn,hamilton2017graphsage,velickovic2018gat}; the published repository narrative highlights a two-layer, 64-hidden-dimension configuration with mean pooling. In the current code, graphs are built densely across windows within each sample, enabling message passing over the full temporal context before graph-level pooling and readout.

\subsection{Training, tuning, and evaluation protocol}

The training utilities unify optimization and reporting across model families. The repository uses binary cross-entropy with logits for five-label prediction, Adam-based optimization, plateau-triggered learning-rate reduction, checkpointing, and saved metric curves. Evaluation reports exact-match accuracy, per-finger averaged accuracy, macro precision, macro recall, macro F1, macro AUROC, and macro AUPRC. The metrics module also stores per-finger scores and task-level curve data, which makes both aggregate and finger-wise analysis available after training.

The paper figure includes a Bayesian-optimization block because tuning is already a real part of the repository workflow rather than a paper embellishment. Current mainline code exposes Optuna-based search utilities for GNN and two-stage training paths, and the open CNN refactor in PR \#56 turns that tuning layer into a latency-aware optimization stage for student size and training settings~\citep{psrpr56cnn2026}. PR \#59 further adds validation-driven threshold tuning and local CPU latency benchmarking, which are relevant for deployment screening even though they are not yet integrated into the main benchmark tables~\citep{psrpr59distill2026}.
